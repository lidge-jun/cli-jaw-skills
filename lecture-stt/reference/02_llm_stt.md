# 02. LLM 기반 STT 원리

## 핵심 차이

| | 전용 STT | LLM STT |
|---|---|---|
| 구조 | Audio → Text (단방향) | Audio + Prompt → Text (멀티모달) |
| 입력 | 오디오만 | 오디오 + 텍스트 프롬프트 |
| 컨텍스트 | ❌ 없음 | ✅ 도메인, 용어, 포맷 지정 가능 |
| 출력 제어 | 고정 | 자유 (요약, 번역, 포맷팅 동시) |
| 가격 모델 | 분당 과금 | 토큰 과금 (오디오도 토큰화) |

## 아키텍처 (Gemini 기준)

```
Audio Waveform
  ↓ Audio Tokenizer (오디오 → 토큰 시퀀스)
  ↓ ─── 합류 ───
  ↓           Text Prompt (토큰)
  ↓ Multimodal Transformer
  ↓ Decoder
  → Text Output
```

### 오디오 토큰화
- Gemini는 오디오를 **내부 토크나이저**로 직접 처리
- 60초 MP3 ≈ 1,500 input tokens (실측)
- 1분당 약 25 토큰 → 1시간 ≈ 1,500 토큰
- 텍스트 토큰과 동일한 어텐션 메커니즘으로 처리

### 왜 빠른가?
- 전용 STT: 30초 청크 단위 sequential 처리
- LLM: 전체 오디오를 한번에 토큰화 → 병렬 처리
- Flash-Lite: 추론 최적화 (quantization, distillation)

## 프롬프트 컨텍스트 효과 (실측)

### 테스트 조건
- 모델: Gemini 3.1 Flash-Lite Preview
- 오디오: 60초 한국어 (AI 스타트업 피칭)

### 결과

| 항목 | generic (맨땅) | context (용어 힌트) |
|------|---------------|-------------------|
| 속도 | 3.37s | **2.09s** (38% ↓) |
| "수정원병원" | ❌ 수종원 병원 | ✅ 수정원 병원 |
| "예비창업패키지" | ❌ 예비 창업 패키지 | ✅ 예비창업패키지 |
| "1만 개" 표기 | 1만 개 | 10,000개 |
| 줄바꿈 | 뭉텅이 | 문장별 ✅ |

### 왜 컨텍스트가 속도도 빠르게 하는가?
- 용어를 미리 주면 디코더의 **탐색 공간이 줄어듦**
- "수정원"이라는 힌트 → "수종원" 후보를 pruning
- 확신도 높은 토큰을 빠르게 생성 → 전체 latency 감소

## 프롬프트 설계 패턴

### 기본형
```
이 오디오를 한국어로 전사해주세요. 전사 텍스트만 출력.
```

### 도메인 특화형
```
이 오디오를 한국어로 전사해주세요.

컨텍스트: [도메인 설명]
관련 용어: [고유명사, 전문용어 나열]
```

### 포맷 지정형
```
이 오디오를 한국어로 전사해주세요.

컨텍스트: [도메인]
관련 용어: [용어들]

출력 형식:
- 문장 단위로 줄바꿈
- 숫자는 아라비아 숫자
- 영어 고유명사는 영어 표기
```

### 응용: 전사 + 요약 동시
```
이 오디오를 전사한 뒤, 끝에 3줄 요약을 추가해주세요.
```

## 🧠 핵심 발견: 전사인가, 변환인가?

LLM STT는 단순 "받아쓰기"가 아니라 **음성을 구조화된 지식으로 변환**한다.

### 실험: 44분 거시경제학 강의 (Gemini 3.1 Flash-Lite)

| 구성 | 시간 | 출력 | 품질 |
|------|------|------|------|
| PDF + 상세 + Non-essential 마킹 | **32s** | **18KB** | 슬라이드별 ✅, KaTeX ✅, 보강 ✅, 비핵심 요약 ✅ 🏆 |
| PDF + 오디오 + 상세 프롬프트 | 20.4s | 13KB | 슬라이드별 구조 ✅, KaTeX ✅, 보강노트 ✅ |
| 오디오 + 프롬프트 (PDF 없음) | 11.6s | 1.2KB | 압축 요약, FOCUS 누락 75% |
| 오디오만 (프롬프트 없음) | 181s | 283KB | 루프 발생 ❌, 구조 없음 |
| 필러 마킹 모드 (filler 유지) | 161s | 297KB | verbatim 전사 ❌, KaTeX 0, 루프 |

### 핵심 인사이트

1. **PDF가 구조의 가이드** — 슬라이드를 넣으면 모델이 "무엇을 기록할지"를 안다.
   PDF 없이는 출력이 23배 커지거나(verbatim), 75% 내용이 누락됨(요약).

2. **프롬프트가 정밀도를 결정** — "보강설명을 이탤릭으로 추가하라"처럼 구체적일수록
   모델은 **교수의 말 vs PDF 내용을 구분**하고, 교수가 설명 안 한 부분까지 보강한다.

3. **필러를 남기면 지능이 꺼진다** — "(uh) 유지하라"고 하면 모델이 verbatim 모드로
   전환되어 구조화 능력 상실. 출력이 23배 폭증 + 루프 발생.
   → LLM STT의 가치 = 필러를 "지우는 것"이 아니라, **음성→지식 변환** 자체.

4. **진짜 전사, 변형 아님** — 적절한 프롬프트를 주면 LLM은 말을 "바꿔쓰기"하는 게 아니라
   교수의 원래 설명을 그대로 전사 + 구조화한다. 전용 STT처럼 날것인데 + 포맷이 붙는 것.

### PDF 구조 가이드 효과 (토큰 비교)

```
PDF + 오디오: 86K input → 3.8K output (압축률 22:1) ← 슬라이드가 골격 제공
오디오만:     66K input → 65K output (압축률 1:1) ← 그냥 다 쏟아냄
```

## 한계

1. **타임스탬프 정밀도** — 전용 STT(단어 단위) 대비 부정확
2. **환각(Hallucination)** — 무음 구간에서 없는 말 생성 가능
3. **화자분리** — Gemini는 미지원 (OpenAI transcribe-diarize만 가능)
4. **매우 긴 오디오** — 8시간 넘으면 청크 분할 필요
5. **필러 마킹 비호환** — "필러를 남겨라" 프롬프트 시 구조화 능력 상실
6. **PDF 없는 긴 강의** — 가이드 없이 30분+ 오디오 넣으면 루프 위험
